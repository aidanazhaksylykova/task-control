# PRD: Мобильное приложение контроля исполнения задач

**Версия:** 1.0
**Статус:** черновик для обсуждения с командой разработки
**Владелец продукта:** руководитель, курирующий несколько проектов

---

## 1. Проблема

Руководитель ведёт несколько параллельных проектов. Основная потеря времени — постоянное ручное уточнение статусов у исполнителей: звонки, чаты, совещания «на каком этапе задача». Нет единой картины по всем проектам, а срывы сроков обнаруживаются постфактум, а не заранее.

## 2. Цель продукта

Заменить ручное уточнение статусов автоматическим контролем: единый реестр задач по всем проектам, самостоятельное обновление прогресса исполнителем и автоматические уведомления при риске или факте срыва срока — одновременно руководителю и исполнителю.

## 3. Цели и метрики успеха

| Цель | Метрика |
|---|---|
| Снизить ручные уточнения статусов | Количество прямых вопросов о статусе вне приложения → стремится к нулю |
| Раньше выявлять риск срыва сроков | Доля задач, помеченных риском/прогнозом до, а не после дедлайна |
| Дать сводную картину без сбора вручную | Время до получения актуального статуса по всем проектам — секунды |
| Повысить долю задач, закрытых в срок | % задач в срок по месяцам — рост к базовому уровню |

## 4. Пользователи и роли

| Роль | Доступ | Основные действия |
|---|---|---|
| Руководитель / куратор | Все проекты, все задачи | Создаёт задачи, назначает исполнителей, видит дашборд по всем проектам, получает эскалации, смотрит аналитику |
| Исполнитель | Только свои задачи | Обновляет статус и % готовности, комментирует, получает напоминания и уведомления о просрочке |
| Наблюдатель (опционально) | Проекты, куда добавлен | Только просмотр — для смежных руководителей или заказчиков |

## 5. Пользовательские сценарии (user stories)

- Как руководитель, я хочу видеть на одном экране, сколько задач в работе, на грани и просрочено по всем проектам, чтобы не собирать статус вручную.
- Как руководитель, я хочу автоматически получать уведомление в момент, когда задача просрочена, а не узнавать об этом позже.
- Как исполнитель, я хочу обновлять прогресс в один тап, без заполнения форм, чтобы руководитель видел статус без моего отдельного отчёта.
- Как руководитель, я хочу видеть прогноз риска по задаче до того, как формально наступит дедлайн, чтобы успеть вмешаться.
- Как руководитель, я хочу видеть, какие задачи блокируют другие, чтобы понимать реальную причину цепных задержек.
- Как исполнитель, я хочу запросить перенос срока с указанием причины, не переписываясь отдельно.
- Как руководитель, я хочу получать еженедельную сводку по почте, не заходя в приложение.

## 6. Модель данных

### 6.1 Проект
| Поле | Тип | Описание |
|---|---|---|
| Название | строка | Название проекта |
| Куратор | ссылка на пользователя | Ответственный руководитель |
| Срок проекта | дата | Плановая дата завершения |
| Статус | вычисляемое | Здоровье проекта: в норме / под угрозой / критический — считается по задачам внутри |

### 6.2 Задача
| Поле | Тип | Описание |
|---|---|---|
| Название | строка | Краткая формулировка задачи |
| Проект | ссылка | К какому проекту относится |
| Постановщик | ссылка на пользователя | Кто создал задачу |
| Исполнитель | ссылка на пользователя | Кто отвечает за выполнение |
| Дедлайн | дата и время | Плановый срок исполнения |
| Приоритет | перечисление | Низкий / средний / высокий |
| Статус | перечисление | Новая / в работе / на проверке / выполнена / просрочена |
| % готовности | число 0–100 | Обновляется исполнителем вручную или по чек-листу подзадач |
| Зависит от | ссылки на задачи | Список блокирующих задач — используется для прогноза риска |
| Причина просрочки | перечисление | Заполняется при закрытии просроченной задачи, из справочника причин |
| История | список событий | Изменения статуса, комментарии, автоматические уведомления |

### 6.3 Комментарий
| Поле | Тип | Описание |
|---|---|---|
| Задача | ссылка | К какой задаче относится |
| Автор | ссылка на пользователя | Кто оставил комментарий |
| Текст | строка | Поддерживает упоминания @пользователь |
| Вложения | список файлов | Фото, документы — например, подтверждение выполнения |

### 6.4 Шаблон повторяющейся задачи
| Поле | Тип | Описание |
|---|---|---|
| Название | строка | Например, «Еженедельный отчёт» |
| Проект | ссылка | К какому проекту привязан |
| Периодичность | перечисление | Ежедневно / еженедельно / ежемесячно / по будням |
| Смещение дедлайна | число | Через сколько часов/дней после создания ставится срок |
| Чек-лист | список пунктов | Копируется в каждую созданную задачу |

## 7. Логика статусов и уведомлений (ядро продукта)

### 7.1 Жизненный цикл задачи
`Новая → В работе → На проверке → Выполнена`, либо автоматический переход в `Просрочена`, если дедлайн наступил, а задача не закрыта.

### 7.2 Триггеры автоматических уведомлений

| Момент | Событие | Получатели |
|---|---|---|
| За 3 дня до дедлайна | Напоминание о приближении срока | Исполнитель |
| За 1 день до дедлайна | Повторное напоминание, если готовность < 100% | Исполнитель |
| В день дедлайна, если не закрыта | Статус меняется на «Просрочена» | Исполнитель и руководитель — одновременно |
| Каждые 24 часа в статусе «Просрочена» | Повторное уведомление | Исполнитель и руководитель |
| На 3-й день просрочки | Эскалация — задача помечается критической | Руководитель проекта / вышестоящий куратор |
| Исполнитель не отреагировал N часов на просрочку | Гибкая эскалация — уведомление уходит выше по цепочке досрочно | Руководитель исполнителя |

Все интервалы должны быть настраиваемыми в параметрах проекта, а не зашиты в код.

### 7.3 Самостоятельное обновление статуса
Исполнитель обновляет % готовности и статус в один тап, без форм. Это заменяет необходимость руководителя запрашивать статус вручную — актуальная картина всегда доступна в реестре.

## 8. Функциональные требования по экранам

### 8.1 Дашборд
- Сводные счётчики по всем проектам: в работе / на грани / просрочено / выполнено за период
- Список проектов с индикатором здоровья (цвет + прогресс-бар)
- План/факт по каждому проекту — отстаёт или опережает график
- Блок «Прогноз риска» — задачи, помеченные риском по темпу прогресса до формального порога «на грани»
- Переход из проекта в отфильтрованный список его задач

### 8.2 Реестр задач
- Список задач по всем проектам с фильтрами: проект, исполнитель, статус, срок
- Сортировка по дедлайну по умолчанию — самые срочные наверху
- Цветовая индикация статуса

### 8.3 Карточка задачи
- Название, проект, исполнитель, описание, текущий статус
- Зависимости: какие задачи блокирует и от каких зависит
- Журнал автоматических уведомлений и изменений статуса
- Комментарии с упоминаниями (@) и вложениями
- Быстрые действия: отметить выполненной, изменить % готовности, запросить перенос срока
- При закрытии просроченной задачи — выбор причины из справочника

### 8.4 Аналитика
- Доля задач, закрытых в срок — по исполнителям и по проектам
- Средняя просрочка в днях
- Рейтинг надёжности исполнителя — % в срок и количество переоткрытых задач
- Тепловая карта загрузки команды — задач в день на каждого исполнителя
- План/факт по проектам за период
- Динамика за выбранный период (неделя / месяц / квартал)

### 8.5 Уведомления
- Единая лента всех автоматических событий: напоминания, просрочки, эскалации, закрытия в срок, еженедельные сводки
- Переход из уведомления сразу в карточку задачи

### 8.6 Ещё (шаблоны и интеграции)
- Шаблоны повторяющихся задач — создание, включение/отключение
- Быстрая постановка задачи: пересылка письма боту, голосовой ввод с автораспознаванием дедлайна и исполнителя
- Интеграции: календарь (двусторонняя синхронизация дедлайнов), импорт задач из Trello / Jira / 1С, Telegram-бот
- Настройка еженедельной авто-сводки на почту

## 9. Нефункциональные требования

- Платформы: iOS и Android (кроссплатформенная разработка — React Native / Flutter)
- Push-уведомления дублируются каналом, который точно доходит вовремя: e-mail или Telegram-бот
- Ролевой доступ: исполнитель не видит чужие задачи, если явно не добавлен наблюдателем
- История изменений хранится не менее 12 месяцев для аналитики
- Время отклика дашборда и реестра — не более 2 секунд при 500+ активных задачах

## 10. Этапы реализации

| Этап | Содержание |
|---|---|
| MVP | Реестр задач, ручное создание/назначение, статусы, авто-уведомления о просрочке, простой дашборд |
| Этап 2 | Аналитика по эффективности, эскалации (в т.ч. гибкая по реакции исполнителя), настраиваемые интервалы напоминаний, комментарии с упоминаниями и вложениями, запрос переноса срока, библиотека причин просрочки |
| Этап 3 | Шаблоны повторяющихся задач, зависимости и прогноз риска, рейтинг надёжности, тепловая карта загрузки, план/факт по проекту, еженедельная авто-сводка, интеграции (календарь, Trello/Jira/1С, Telegram-бот), автопостановка из писем и голосовой ввод |

## 11. Открытые вопросы

- Какой канал уведомлений приоритетный: push, e-mail или Telegram?
- Нужна ли интеграция с уже используемыми системами (Jira, Trello, 1С, корпоративная почта)?
- Кто утверждает эскалацию на 3-й день просрочки — фиксированный руководитель или настраиваемая цепочка?
- Нужны ли подзадачи и чек-листы внутри задачи для более точного расчёта % готовности?
